# Academy Hub Runtime Runbook

Operational guide for running the Academy Hub API in two scenarios:

- `local`: local backend + ngrok for external callbacks
- `render`: local backend in production-like mode, pointing webhook URLs to Render

This runbook exists to avoid confusion between:

- `localhost`
- `127.0.0.1`
- ngrok
- Render

## Commands

### Local scenario

```bash
npm run academy:local
```

What this command does:

- starts the API on port `8081`
- starts an ngrok tunnel for that port
- injects the public ngrok URL into:
  - `NGROK_URL`
  - `PROD_URL`
  - `EVOLUTION_WEBHOOK_URL`
- makes Evolution register the webhook pointing to the local ngrok URL

When to use:

- local development
- webhook validation
- debugging `SEND_MESSAGE`, `MESSAGES_UPDATE`, and ledger persistence
- reproducing the full flow without depending on Render

### Render scenario

```bash
npm run academy:render
```

What this command does:

- starts the API on port `8081`
- starts an ngrok tunnel for that port
- runs the API with `NODE_ENV=production`
- injects the Render URL into:
  - `PROD_URL`
  - `EVOLUTION_WEBHOOK_URL`
- keeps ngrok available for manual access to the local process, but the webhook target is the Render URL

When to use:

- validating the same API with production configuration
- comparing local behavior with deployed behavior
- keeping the backend local while matching the production URL contract

Important:

- this command does not start the Render deployment
- it runs the same application locally with production URLs
- if you need the actual Render deployment, use the deployed Render URL directly

## Environment assumptions

### Evolution local

For the local lab, Evolution should remain reachable at:

- `http://127.0.0.1:3000`

In the Evolution project, the app callback URL is still configured by the application itself when the instance connects.

### Academy Hub local

The launcher above runs the API on port:

- `8081`

This avoids collisions with the local Evolution process and keeps the topology consistent with the tests we ran.

Optional overrides:

- `ACADEMY_HUB_PORT` to change the local port
- `ACADEMY_HUB_RENDER_URL` to change the production URL used in render mode

## What each URL means

### `EVOLUTION_API_URL`

The Evolution endpoint the Academy Hub calls for:

- connecting an instance
- sending messages
- configuring the instance webhook

In this local lab:

- `http://127.0.0.1:3000`

### `NGROK_URL`

The public URL generated by the ngrok tunnel.

In local mode, this becomes the real webhook target for Evolution.

### `PROD_URL`

The base URL used when the API runs with `NODE_ENV=production`.

In render mode, it points to:

- `https://school-management-api-76ef.onrender.com`

## How to interpret status

The transport ledger should be read as:

- `queued`: local send request created
- `accepted_by_evolution`: Evolution accepted the request
- `server_ack`: Evolution received confirmation from the WhatsApp server
- `delivered`: WhatsApp confirmed device delivery
- `read`: recipient read the message
- `failed`: send failure
- `deleted`: message removed

If a message stays in:

- `accepted_by_evolution`

and does not move to:

- `server_ack`
- `delivered`

the issue is no longer in the front end or in the Academy Hub controller. The focus moves to the Evolution -> WhatsApp -> delivery path.

## Recommended test flow

### 1. Start Evolution local

Use the Evolution local instance on port `3000`.

### 2. Start the Academy Hub

Use one of the commands above.

### 3. Connect the WhatsApp instance

When the application connects the instance, the webhook will be stored using the URL for the selected mode.

### 4. Send a small message

Prefer:

- short text
- or a simple PDF

### 5. Check the ledger

Validate `GET /api/notifications/transport-logs` for:

- `provider_message_id`
- `provider_status`
- `server_ack_at`
- `delivered_at`
- `read_at`
- `raw_send_response`
- `raw_last_webhook_payload`

## Common issues

### `POST /api/webhook/whatsapp` returns 404 in ngrok

This usually means the tunnel is pointing to the wrong process.

On Windows, this often happens when:

- `localhost` lands on a different listener
- `127.0.0.1` and `localhost` do not resolve to the same process

### Message stays on one tick

Expected signs:

- Evolution accepts the request
- Academy Hub records `accepted_by_evolution`
- no `server_ack` or `delivered` appears

This points to a delivery or status-tracking issue after initial acceptance.

### Webhook arrives but the ledger does not update

Check:

- `EVOLUTION_WEBHOOK_URL`
- events subscribed in the instance
- `WhatsappTransportLog` persistence

## Practical rule

- use `academy:local` when you want to reproduce the full chain with a local webhook through ngrok
- use `academy:render` when you want to validate the production contract without editing `.env` manually
